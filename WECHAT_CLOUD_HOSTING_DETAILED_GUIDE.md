## GitHub 授权

### 2.1 授权步骤

**步骤 1：点击授权链接**

在云托管界面中，找到红色提示文字：
```
代码权限未授权或授权过期。点击授权
```

点击蓝色的「点击授权」链接。

**步骤 2：GitHub 授权页面**

会跳转到 GitHub 授权页面，点击「Authorize」或「授权」按钮。

**授权权限说明：**
- 读取仓库信息（选择代码仓库）
- 读取仓库内容（获取代码和 Dockerfile）
- 创建 Webhook（自动触发部署）

**步骤 3：授权成功**

授权成功后，会自动跳转回云托管界面，红色提示消失。

## 选择代码

### 3.1 配置代码仓库

**步骤 1：选择仓库**

1. 在「代码仓库」下拉框中，点击右侧的下拉箭头
2. 从列表中选择你的 GitHub 仓库
   - 包含 `server/Dockerfile` 的仓库
   - 例如：`skin-analysis-server`

**如果仓库不在列表中：**
- 检查是否授权了正确的 GitHub 账号
- 确认仓库是公开的，或者授权账号有访问权限
- 刷新页面重新加载仓库列表

### 3.2 选择分支

**步骤 1：选择分支**

1. 在「分支」下拉框中，选择主分支
   - 通常是 `main` 或 `master`
   - 建议使用 `main`（更现代）

**步骤 2：确认 Dockerfile 存在**

确保选择的分支中存在 `Dockerfile` 文件：
- 位置：`Dockerfile`（根目录）或 `server/Dockerfile`
- 如果不在根目录，需要在「目标目录」中指定

## 配置环境变量

### 4.1 打开环境变量开关

1. 在「高级设置」区域，找到「环境变量」
2. 点击开关，将其从灰色（关闭）切换为绿色（开启）
3. 右侧会显示「JSON」标识

### 4.2 添加环境变量

**方式一：逐个添加（推荐用于配置少量变量）**

1. 在环境变量区域，找到输入框
2. 在左侧输入框（Key）中输入变量名
3. 在右侧输入框（Value）中输入变量值
4. 点击右侧的 `+` 按钮添加
5. 重复上述步骤，添加所有必需的环境变量

**方式二：JSON 格式（推荐用于配置大量变量）**

1. 点击环境变量开关右侧的「JSON」标识
2. 在弹出的文本框中，粘贴以下 JSON 配置：

  "NODE_ENV": "production",
  "COZE_API_KEY": "8f38a8d0-b6d4-481c-8e15-b93bb30c7444",
  "COZE_MODEL": "doubao-vision",
  "COZE_SUPABASE_URL": "https://pacqfzvxkiobtxbjubil.supabase.co",
  "COZE_SUPABASE_ANON_KEY": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJpc3MiOiJzdXBhYmFzZSIsInJlZiI6InBhY3FmenZ4a2lvYnR4Ymp1YmlsIiwicm9sZSI6ImFub24iLCJpYXQiOjE3NzMzODM1ODYsImV4cCI6MjA4ODk1OTU4Nn0.CZoSjahWflEy3vhj_Ya9ddYhie9BtmvRxhNaN8JdZs4",

## 部署发布

### 5.1 确认配置

在点击「下一步」之前，再次确认以下配置：

| 配置项 | 正确值 | 状态 |
|--------|--------|------|
| 选择方式 | 绑定 GitHub 仓库 | [ ] 已确认 |
| 代码仓库 | 你的 GitHub 仓库名 | [ ] 已确认 |
| 分支 | main 或 master | [ ] 已确认 |
| 端口 | 80 | [ ] 已确认 |
| 环境变量 | 所有必需变量已配置 | [ ] 已确认 |
| 目标目录 | 空白（如果 Dockerfile 在根目录）或 `server` | [ ] 已确认 |
| Dockerfile 文件 | 有 | [ ] 已确认 |
| Dockerfile 名称 | Dockerfile | [ ] 已确认 |

### 5.2 开始部署

**步骤 1：点击「下一步」**

1. 检查所有配置无误
2. 点击右下角的「下一步」按钮
3. 进入「部署发布」步骤

**步骤 2：确认部署配置**

在「部署发布」页面，确认以下信息：

- **环境名称**：`prod`
- **服务名称**：自动生成（例如：`skin-detection-server`）或手动输入
- **版本名称**：自动生成（例如：`v1`）或手动输入
- **实例规格**：
  - CPU：0.25 核（免费额度足够）
  - 内存：512MB（免费额度足够）
  - 实例数：1（自动扩缩容）

**步骤 3：点击「部署」**

1. 检查部署配置无误
2. 点击右下角的「部署」按钮
3. 部署开始，进入部署状态页面

### 5.3 部署状态

**部署过程分为以下几个阶段：**

1. **代码拉取**（约 30 秒）
   - 从 GitHub 拉取代码
   - 状态显示：⏳ 代码拉取中

2. **镜像构建**（约 2-5 分钟）
   - 根据 Dockerfile 构建镜像
   - 安装依赖和构建应用
   - 状态显示：⏳ 镜像构建中

3. **服务部署**（约 1-2 分钟）
   - 创建容器实例
   - 启动应用
   - 健康检查
   - 状态显示：⏳ 服务部署中

4. **部署完成**
   - 服务启动成功
   - 状态显示：✅ 部署成功

**总耗时：约 3-8 分钟**

### 5.4 查看部署日志

**在部署过程中，可以查看实时日志：**

1. 在部署状态页面，点击「查看日志」
2. 查看详细的部署日志
3. 如果出现错误，日志会显示具体的错误信息

**常见部署错误：**

| 错误信息 | 原因 | 解决方案 |
|----------|------|----------|
| `Dockerfile not found` | Dockerfile 文件不存在 | 检查 Dockerfile 是否在正确的位置 |
| `Build failed` | 构建失败 | 查看构建日志，检查代码或依赖 |
| `Health check failed` | 健康检查失败 | 检查应用是否正确启动 |
| `Port already in use` | 端口被占用 | 检查端口配置是否正确 |

## 常见问题

### Q1: GitHub 授权失败怎么办？

**问题：** 点击授权后，仍然显示「代码权限未授权」

**解决方案：**

1. 检查 GitHub 账号是否正确
2. 确认授权页面是否点击了「Authorize」
3. 尝试刷新页面
4. 重新点击授权链接

---

### Q2: 找不到我的 GitHub 仓库？

**问题：** 代码仓库下拉列表中没有我的仓库

**可能原因：**

1. 仓库是私有的，但没有授权 GitHub 账号
2. 仓库属于其他组织或团队
3. 仓库刚刚创建，还未同步

**解决方案：**

1. 确认仓库是否为公开
2. 如果是私有仓库，确保授权的 GitHub 账号有访问权限
3. 刷新页面，重新加载仓库列表
4. 检查是否授权了正确的 GitHub 账号

---

### Q3: Dockerfile 构建失败怎么办？

**问题：** 部署时出现「Build failed」错误

**排查步骤：**

1. 查看构建日志，找到具体的错误信息
2. 检查 Dockerfile 是否在正确的位置
3. 检查 package.json 是否有启动脚本
4. 检查依赖是否完整

**常见错误及解决方案：**

| 错误信息 | 原因 | 解决方案 |
|----------|------|----------|
| `Cannot find module` | 依赖未安装 | 检查 package.json，确保所有依赖都已声明 |
| `Command not found` | 命令不存在 | 检查 Dockerfile 中的命令是否正确 |
| `Permission denied` | 权限不足 | 检查 Dockerfile 中的文件权限 |
| `ENOENT` | 文件不存在 | 检查文件路径是否正确 |

---

### Q4: 部署成功后，服务无法访问？

**问题：** 服务部署成功，但 curl 测试失败

**排查步骤：**

1. 检查服务地址是否正确
2. 查看服务日志，确认应用是否启动成功
3. 检查端口配置是否正确（应该是 80）
4. 检查环境变量是否配置完整
5. 查看监控数据，确认服务是否正常运行

---

### Q5: 小程序仍然出现网络错误？

**问题：** 部署成功后，小程序仍然出现 `ERR_CONNECTION_RESET`

**解决方案：**

1. **清除缓存**：在开发者工具中清除全部缓存
2. **重新编译**：重新编译小程序
3. **检查配置**：确认 `config/index.ts` 中的 `PROJECT_DOMAIN` 是否正确
4. **测试服务**：使用 curl 测试服务地址是否可访问
5. **查看日志**：查看服务日志，确认是否收到请求

---

### Q6: 如何更新服务？

**场景：** 修改了代码，需要重新部署

**更新步骤：**

1. **提交代码**：将修改提交到 GitHub
2. **触发部署**：
   - 方式一：在云托管控制台，点击「部署」→「选择版本」→「新建版本」
   - 方式二：推送代码后，自动触发部署（如果配置了 Webhook）

**自动部署配置：**

1. 进入服务详情
2. 点击「流水线」
3. 配置自动触发规则：
   - 推送到 main 分支时自动部署
   - 创建 PR 时自动部署测试环境

---

### Q7: 如何回滚到之前的版本？

**场景：** 新版本部署后出现问题，需要回滚

**回滚步骤：**

1. 进入服务详情
2. 点击「版本」
3. 找到之前的版本（例如：v1）

5. 确认回滚

---

### Q8: 如何扩容服务？

**场景：** 用户量增加，需要提升服务性能

**扩容步骤：**

1. 进入服务详情
2. 点击「配置」
3. 修改「实例规格」：
   - CPU：0.25 核 → 0.5 核 → 1 核
   - 内存：512MB → 1GB → 2GB
4. 修改「实例数」：
   - 从 1 增加到 2 或更多（支持自动扩缩容）
5. 点击「保存」
6. 等待服务重启（约 1-2 分钟）
